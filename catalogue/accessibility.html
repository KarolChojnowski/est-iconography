Implementation guidance

Accessible icon use

Choose the SVG treatment from its context. Visible text, controls and status messages own the meaning—not the artwork or canonical asset name.

Guardrails

Common mistakes to avoid

Do not name decorative SVGs

An icon beside matching text can create a duplicate announcement when both are exposed.

Do not leave icon controls unnamed

A button with only an SVG still needs a specific accessible name on the button.

Do not rely on colour or shape alone

Keep visible text for errors, warnings, success and other important states.

Do not make the SVG interactive

Use a native button or link for focus, keyboard behaviour and activation.

Before release

Implementation checklist

  • Decide whether the icon is decorative or informative.
  • Give every icon-only button or link a specific accessible name.
  • Hide decorative and control-owned SVGs with aria-hidden="true" and focusable="false".
  • Give standalone informative SVGs role="img" and a contextual name.
  • Keep visible text for status, warning, error and success meaning.
  • Test the containing control’s keyboard focus rather than the SVG.
  • Inspect the result in a browser accessibility tree or with a screen reader.
